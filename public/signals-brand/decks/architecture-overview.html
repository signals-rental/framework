SIG-AO03Infrastructure KernelRev A

Layer 1: Infrastructure Kernel

The foundation that every other layer depends on. Invisible when it works, catastrophic when it doesn’t.

Queue & Job Infrastructure
Redis primary (db0–db3), database fallback. Six named queues: default, webhooks, notifications, exports, imports, mail. Horizon dashboard. Batch and chain patterns.
Caching Strategy
Redis cache-aside pattern. Tag-based invalidation. Settings cached indefinitely, navigation 1hr, widgets 5min. Invalidate on write, never rely on TTL alone.
Settings & Administration
Runtime configuration stored in database. settings('group.key') accessor. Grouped definitions. Indefinite cache with tag invalidation. Encrypted credentials.
Observability
Laravel Nightwatch APM. Request tracing, slow query detection, error tracking. Health check endpoints for database, Redis, S3 connectivity.
SSO & Enterprise Auth
Email/password native. Socialite OAuth (GitHub, Google). SAML 2.0 / OIDC for enterprise. Sanctum API tokens. TOTP-based MFA. JIT user provisioning.
File & Attachment System
S3-only storage. UUID-based paths. Signed temporary URLs. Polymorphic attachments table. Optional ClamAV scanning. Thumbnails via Intervention Image.
SIG-AO04Data & SchemaRev A

Layer 2: Data & Schema

The metadata backbone. Field definitions, custom attributes, access control, and API conventions that every surface consumes.

Field Registry & Schema Engine
Three-source merge: core fields (migrations), computed fields (code), custom fields (EAV). Five consumers: Custom Views, Reporting, Import/Export, API, Plugins.
Custom Fields (EAV)
17 field types. Typed value columns. Field groups, validation rules, visibility rules. Auto-copy between entities. Queryable via Ransack: q[cf.field_name_eq]=value
Permissions & Authorisation Engine
Four layers: UI areas, actions, cost visibility, store scoping. resource.action naming. Five built-in roles. Owner has implicit all-access. Gate checks in action classes.
API Architecture
REST with CRMS drop-in compatibility. Spatie Data DTOs for input validation + response serialisation. Ransack-compatible query engine. Sanctum bearer tokens.

SIG-AO05Money & LocalisationRev A

Layer 3: Money & Localisation

Every monetary operation flows through these three engines. Lossless arithmetic from rate calculation through to invoice totals.

Rate Engine
Calculation strategies: period-based, usage-based, fixed, hybrid. Composable modifiers: multiplier (tiered), factor (volume). 12 CRMS-compatible presets.
Base Price
Multi-Currency & Tax Engine
brick/money value objects. Integer minor-unit storage. Temporal exchange rates. Tax zones × categories → rates. Per-line RationalMoney arithmetic.
Taxes & Currency
Discount & Pricing Rules
Four-stage pipeline: price categories → discounts → surcharges → manual override. Highest discount wins. Non-discountable flags. Deal pricing.

Data flows left to right. Rate Engine produces the base price. Currency & Tax converts. Discount rules apply final adjustments.

Layer 4: Operational Core

The heart of the rental system. Quote-to-return lifecycle, real-time availability, shortage detection, and kit management.

Opportunity Lifecycle
Two-axis state model (state × status). Verbs event sourcing. Quote versioning. Line-item granularity.
demands
dispatch
Availability Engine
Pre-calculated snapshots. Unified demand table. Five behavioural phases. tstzrange/GiST indexing.
Serialised Containers
Template-driven kits. Strict vs flexible packing. Dissolve-on-dispatch. Scanning integration.

Computed at query time. Resolver registry: internal transfer, substitute, sub-hire. Virtual stock. Resolution records flow back to Opportunity.

Demands flow down from Opportunity to Availability. Negative slots trigger Shortage Resolution. Resolutions feed back to Opportunity.

SIG-AO07Event BusRev A

The Event Bus: One Event, Five Reactions

A cross-cutting concern that connects every layer. Every domain event flows through one dispatcher. Five independent consumers react.

Event Bus
Audit Log
Synchronous. Before/after JSONB snapshots. IP address, user agent. Append-only. Configurable retention.
Webhooks
Queued. HMAC-SHA256 signed. Exponential backoff (6 retries, 12 hours). Auto-disable after 3 days of failures.
Workflows
Queued or synchronous. Handler-based step chains. Conditions, delays, halts. Loop protection.
Notifications
Queued. Multi-channel (email, SMS, in-app, push). Three-layer preference resolution. Template integration.
Approval Chains
Queued or synchronous. Two-event pattern. Quorum (any/all). Delegation, escalation, expiry.

Each consumer operates independently. They share the bus but have no knowledge of each other.

Layer 6: Service Systems

Eight specialised engines that make the platform production-ready and operationally complete.

Notification Engine
Multi-channel messaging. Three-layer resolution. Template system. Delivery tracking.
Document Template Engine
Blade templates per store. PDF rendering. Field browser. Public sharing.
Import/Export Engine
SQLite-staged migration. CRMS pre-built mappings. Dry-run validation.
Scanning Layer
USB, QR, RFID, manual entry. Three-layer pipeline: input → resolve → context.
Maintenance Engine
Time + usage inspections. Auto-quarantine. Certification tracking. TCO analysis.
Scheduling Engine
Crew, vehicle, facility, equipment. Conflict detection. Route management.
Approval Chain Engine
Multi-stage chains. Delegation, escalation. Internal and external approvers.
Workflow Engine
Event-driven automation. Handler registry. Conditions, delays, halts.
SIG-AO09Intelligence & ViewsRev A

Layer 7: Intelligence & Views

The data consumption layer. Every system below generates data; this layer makes it visible, queryable, and actionable.

Reporting Framework
Aggregated queries over shared field registry. Dimensions and measures. Comparison periods (YoY, MoM). Scheduled delivery.
Custom Views
Saved list configurations: columns, filters, sort. Personal/shared/system visibility. Conditional formatting. Dynamic tokens.
Search System
PostgreSQL tsvector/GIN full-text search. Global search modal. Entity-type scoping. Indexed across all key fields.
Dashboard System
Section layout. Widget registry (core + plugins). Real-time updates via Reverb WebSocket. Shared dashboards.
SIG-AO10Extension SurfaceRev A

Layer 8: Plugin System

Every layer below exposes extension points. The Plugin System provides the SDK, lifecycle management, and mediated access that third-party developers use to extend every surface.

YAML Manifest

Each plugin ships a signals.json manifest declaring name, version, permissions, events, custom fields, routes, and settings. The SDK validates this at discovery time.

Mediated Data Access

Plugins never import core models directly. All data access goes through the mediated API — query builders exposed by the SDK. This prevents breaking changes when core schema evolves.

Table Prefix

Plugin tables use the plugin_ prefix. No plugin-owned migrations on core tables. Extend via custom fields and JSONB columns instead.

PluginRegistrar API

Fluent registration interface for: routes, events, actions, custom fields, widgets, notifications, reports, nav items, permissions, settings.

Lifecycle

Discover → Validate → Register → Activate → Boot → Disable

SIG-AE02The Core QuestionRev A

Why Not Just Count at Query Time?

The naive approach calculates availability on every request: count stock, subtract overlapping demands. It works — until it doesn’t.

Computed on demand
Every page load runs: SELECT stock - SUM(demands) WHERE dates overlap.

At 50 products × 30 days, that’s 1,500 aggregate queries per grid view. Each query scans the demands table, joins stock levels, and groups by date.

Under concurrency, two users confirming bookings simultaneously can both see “1 available” and both succeed — creating an undetected shortage.
Pre-calculated snapshots
Availability is written to a snapshot table when demands change. Reading is a direct index scan: WHERE product + warehouse + slot_start.

At 50 products × 30 days, that’s 1,500 rows from a single indexed table. No joins, no aggregation, no computation.

Writes acquire a lock per product/warehouse, so concurrent confirmations are serialised. No silent overbooking.

The trade-off: writes are more expensive. But reads vastly outnumber writes in rental software.

SIG-AE03Demand ModelRev A

One Table for Every Demand Source

Bookings, quarantines, transfers, plugin demands — they all write to the same unified demands table. The engine doesn’t know or care what created the demand.

Why this matters

When the engine recalculates, it runs one query: give me all active demands for this product at this warehouse in this time range. It gets back everything — bookings, maintenance windows, transit holds, plugin demands — in a single result set.

Adding a new demand source (say, a future “loans” module) requires zero changes to the engine. Register the source, write demands to the table, and the engine picks them up automatically.

Sentinel dates

Open-ended demands (overdue items with no return date) use 2199-01-01 instead of null. This eliminates null handling in every time-range query: starts_at <= :end AND ends_at >= :start. Clean range scans, always.

SIG-AE04Demand SourcesRev A

Where Demands Come From

A demand source is anything that claims stock for a time period. The engine ships with three built-in sources, and plugins can register more.

Built-in sources

Opportunity Line Items — the primary demand. A booking with 5 speakers from March 10–17 creates a demand for 5 units across that range. Each line item is its own demand record.

Quarantines — maintenance holds, damage inspections, cleaning windows. A quarantine on 2 units for 3 days creates a demand identical in shape to a booking.

Transfers — stock movements between warehouses. Creates two demands (source outbound + destination inbound) to account for transit time.

Plugin sources

Any plugin can register a new demand source. A “loans” module, a “sub-hire” tracker, a “seasonal reservation” system — they all write to the same demands table with their own source_type. The engine picks them up with zero changes.

SIG-AE05Behavioural PhasesRev A

States Are Infinite. Phases Are Five.

Every tenant configures different state names. “Confirmed”, “Deposit Paid”, “Awaiting Collection” — these all mean the same thing to the engine: stock is reserved. Phases are the fixed contract between configurable state machines and availability calculation.

Draft
Demand exists but is not active. No impact on availability.
enquiry, quote, provisional, scoping
Committed
Demand is active. Stock is reserved.
confirmed, approved, deposit paid, pending
Operational
Demand is active. Stock is out the door.
dispatched, on hire, partially returned
Closed
Demand released. Stock returns after turnaround.
returned, checked, settled
Void
Demand erased. Immediate release, no turnaround.
cancelled, rejected

Default states have locked phase mappings. Custom states can be assigned to any phase. Phases cannot be added or removed.

SIG-AE06Snapshot ModelRev A

The Snapshot Table Is the Read Model

One row per product × warehouse × time slot. The available column is the answer. No computation at query time.

Resolution is a setup-time decision

The slot width determines how many rows exist. It’s chosen at installation and cannot be changed without a full rebuild.

SIG-AE07RecalculationRev A

How Snapshots Stay In Sync

When a demand changes, recalculation targets only the affected product/warehouse/time range. The pipeline runs synchronously for single-entity changes (< 300ms) and asynchronously for bulk operations.

01
Event Received
A demand is created, updated, phase-transitioned, or released. Carries product_id, warehouse_id, and affected time range.
02
Gather Stock
Query stock level for this product at this warehouse, accounting for time-varying changes (scheduled purchases, planned write-offs).
03
Gather Demands
One query against the unified demands table: all active demands for this product/warehouse overlapping the affected range. All sources returned.
04
Strategy Hook
Optional plugin hook: modify the demand set before calculation (buffer stock, priority allocation). Default: pass-through.
05
Calculate Slots
For each slot in the range: sum overlapping demand quantities, subtract from stock. Produce available count and demand_breakdown JSON.
06
Upsert Snapshots
Write snapshot rows with a lock on (product_id, warehouse_id). Update daily summaries for sub-daily installations.
07
Detect Shortages
If any slot is newly negative: emit shortage_detected. If a previously negative slot recovers: emit shortage_resolved.
08
Broadcast
Emit availability_changed to WebSocket channels, webhook queue, and event log. Notify kit dependents if this product is a component.
SIG-AE08StrategiesRev A

Availability Is Pluggable

The recalculation pipeline is fixed, but how availability is derived from stock and demands is defined by a strategy. The system ships a default. Plugins can replace it.

System default

Simple subtraction: available = stock − active demands. For each time slot, sum all active demand quantities and subtract from total stock. This is correct for most rental businesses.

Plugin strategies

A plugin registers its own strategy to change the calculation. The strategy receives the full demand set and stock level, and returns the availability figure per slot.

Examples
  • → Buffer stock — always hold back N units as safety margin
  • → Overbooking — allow N% more bookings than physical stock
  • → Channel allocation — reserve portions of stock for specific sales channels
  • → Priority tiers — VIP customer demands take precedence over standard
// Strategy interface (simplified)

interface AvailabilityStrategy
{
  public function calculate(
    int $totalStock,
    Collection $demands,
    CarbonPeriod $slot
  ): int; // → available
}

SIG-AE09Extension PointsRev A

How Plugins Interact With the Engine

The engine exposes four extension surfaces. Plugins can use any combination without modifying core code.

Events (read-only signals)

The engine emits events at every state change: demand_created, availability_changed, shortage_detected, shortage_resolved. Listeners react — send alerts, update dashboards, trigger webhooks — but cannot modify the calculation.

Hooks (pipeline intercepts)

Hooks modify data mid-pipeline. The strategy hook (step 4) is the most powerful — it can change how availability is calculated. Other hooks allow demand filtering and post-calculation transformations.

APIs (query & write)

REST and internal APIs for reading availability (snapshot queries) and writing demands (creating/updating demand records). All APIs enforce phase validation and trigger recalculation automatically.

Demand registration

Plugins register new demand source types at boot. The engine validates the source_type exists before accepting demand writes. No schema changes required — all sources share the same table.

Events
demand_created · demand_updated · phase_transitioned
availability_changed · shortage_detected · shortage_resolved

Events are fire-and-forget. Hooks are synchronous and can modify data. APIs validate and trigger recalculation.

SIG-AE10Line-Item GranularityRev A

Why Track at the Line Item, Not the Order?

A large event booking has 40 line items. 38 come back on Friday; 2 remain out. Under an order-level model, all 40 items show as unavailable until the entire order closes. Under a line-item model, the 38 returned items release immediately.

The ceiling principle

The opportunity’s phase sets the maximum phase its line items can reach. When the opportunity is draft, all line items are draft. When it’s committed, line items activate. Within that ceiling, line items move independently.

Override mechanism

Cancelling or closing an opportunity forces all its line items to that phase — regardless of their individual state. This is how a single cancellation action releases all demands.

SIG-AE11KittingRev A

Why Kits Have No Snapshot Rows

A kit’s availability depends on its components. Components are shared across kits and standalone bookings. Maintaining snapshot rows for kits would require cascade recalculation — every time any component changes, every kit containing it would need rebuilding.

The alternative: calculate at read time

A kit with 15 components across 30 days reads at most 450 rows from the snapshot table. The arithmetic is trivial. This eliminates cascade propagation, deduplication of cascade triggers, and eventual consistency between component and kit snapshots.

When a kit is booked

Demands are created for each component at the required quantities. The kit itself generates no demand. Demand is always tracked at the leaf product level.

Worked Example: PA System Package
Speakers (need 2/kit, 12 available)FLOOR(12/2) = 6
Mixer (need 1/kit, 4 available)FLOOR(4/1) = 4
XLR Cables (need 4/kit, 30 available)FLOOR(30/4) = 7
Stands (need 2/kit, 9 available)FLOOR(9/2) = 4

SIG-AE12TransfersRev A

The Dual-Demand Transfer Model

A stock transfer between warehouses creates two demand records. During transit, the units are unavailable at both locations. This is the correct behaviour — stock in a van can’t be hired from either warehouse.

Source warehouse

A demand is created from departure time onward. Availability decreases at the source from the moment the stock leaves.

Destination warehouse

Stock only becomes available after arrival plus check-in time. Before that, it exists in transit and is unavailable.

Completion

When the transfer completes (stock arrives and is checked in), the source demand closes and the destination stock level increases. Both warehouses’ snapshots recalculate independently.

Each warehouse recalculates independently. Cross-warehouse totals aggregate at read time.
